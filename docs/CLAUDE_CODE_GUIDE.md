# Claude Code 사용 가이드

이 문서는 비개발자가 Claude Code를 사용하여 웹사이트를 직접 수정하는 방법을 안내합니다.

## 1. Claude Code란?

Claude Code는 **채팅처럼 한국어로 대화하면서 코드를 수정할 수 있는 AI 도구**입니다.
프로그래밍을 몰라도, 원하는 변경 사항을 자연어로 설명하면 Claude가 코드를 수정해 줍니다.

예를 들어:
- "사이트 이름을 '나의 포트폴리오'로 바꿔줘"
- "글 목록에서 날짜 형식을 '2024년 1월 1일'로 변경해줘"
- "푸터에 인스타그램 링크를 추가해줘"

---

## 2. 터미널 기초

Claude Code는 **터미널**(명령어 입력 창)에서 동작합니다.
처음 접하면 낯설 수 있지만, 몇 가지만 알면 충분합니다.

### 터미널이란?

컴퓨터에 **글자로 명령을 내리는 창**입니다. 마우스로 클릭하는 대신, 키보드로 명령어를 입력합니다.
Windows에서는 "PowerShell", Mac에서는 "터미널"이라고 부릅니다.

### 터미널 여는 법

**Windows:**
1. 화면 아래 작업 표시줄의 **검색창**에 `PowerShell` 입력
2. "Windows PowerShell"이 검색되면 클릭하여 실행

**Mac:**
1. `Cmd + Space`를 눌러 Spotlight 검색 열기
2. `Terminal` 입력 후 Enter

### 터미널 화면 읽는 법

터미널을 열면 이런 모습이 보입니다:

```
PS C:\Users\홍길동>
```

- `PS` — PowerShell이라는 뜻
- `C:\Users\홍길동` — 지금 내가 보고 있는 폴더 위치
- `>` 뒤에 커서가 깜빡임 — 여기에 명령어를 입력

### 꼭 알아야 할 명령어 2개

**1. `cd` — 폴더 이동 (Change Directory)**

```
cd C:\Users\홍길동\Documents
```

파일 탐색기에서 폴더를 더블클릭하여 들어가는 것과 같습니다.

> **폴더 경로를 모르겠다면?** 파일 탐색기에서 원하는 폴더를 열고,
> 상단 주소 표시줄을 클릭하면 경로가 텍스트로 바뀝니다. 그걸 복사하면 됩니다.

**2. `ls` — 현재 폴더의 파일 목록 보기**

```
ls
```

파일 탐색기에서 폴더를 열었을 때 보이는 파일 목록과 같습니다.

### 터미널에서 복사/붙여넣기

| 동작 | Windows PowerShell | Mac 터미널 |
|------|-------------------|-----------|
| 복사 | 텍스트 드래그 후 `Ctrl+C` | 텍스트 드래그 후 `Cmd+C` |
| 붙여넣기 | **마우스 오른쪽 클릭** 또는 `Ctrl+V` | `Cmd+V` |

> 이 문서에 나오는 명령어는 직접 타이핑하지 않아도 됩니다.
> 문서에서 복사한 뒤 터미널에 붙여넣고 Enter를 누르면 실행됩니다.

## 3. 사전 준비 (최초 1회)

처음 한 번만 설정하면 됩니다. 이후에는 바로 사용할 수 있습니다.

### Step 1: 필수 프로그램 설치

아래 두 프로그램을 설치합니다. 이미 설치되어 있다면 건너뛰세요.

**Git for Windows (Windows만 해당):**
1. [gitforwindows.org](https://gitforwindows.org/) 접속
2. "Download" 클릭하여 설치 파일 다운로드
3. 설치 프로그램 실행 → 모든 옵션 기본값으로 "Next" 클릭 → "Install"

> **Git이 뭔가요?** 코드의 변경 이력을 관리하는 도구입니다.
> Claude Code가 코드를 저장하고 사이트에 반영할 때 사용합니다. 한 번 설치하면 됩니다.

**Node.js:**
1. [nodejs.org](https://nodejs.org/) 접속
2. **LTS** 버전(왼쪽 초록 버튼) 다운로드
3. 설치 프로그램 실행 → 모든 옵션 기본값으로 "Next" 클릭 → "Install"

> **Node.js가 뭔가요?** 이 웹사이트를 내 컴퓨터에서 실행하기 위해 필요한 프로그램입니다.
> 한 번 설치하면 됩니다.

**설치 확인:** 설치가 잘 되었는지 확인하려면 PowerShell을 **새로 열고** 아래 명령어를 각각 입력하세요.
버전 번호가 나오면 성공입니다.

```
git --version
```
```
node --version
```

### Step 2: Claude Code 설치

1. PowerShell을 **관리자 권한으로** 실행합니다:
   - 작업 표시줄 검색창에 `PowerShell` 입력
   - "Windows PowerShell"을 **마우스 오른쪽 클릭** → "관리자 권한으로 실행" 선택
2. 아래 명령어를 복사하여 PowerShell에 붙여넣고 Enter:

```
irm https://claude.ai/install.ps1 | iex
```

3. 설치가 완료되면 PowerShell을 닫았다가 다시 열어주세요.

**Mac / Linux:**
```
curl -fsSL https://claude.ai/install.sh | bash
```

> 설치 후 자동 업데이트를 지원하므로 별도 관리가 필요 없습니다.

### Step 3: Anthropic 구독

1. [claude.ai](https://claude.ai)에 접속
2. 계정이 없다면 회원가입
3. Pro 플랜 이상 구독
   - Pro ($17/월, 연간 결제 기준): 일반 사용에 충분
   - Max ($100/월~): 더 많은 사용량이 필요할 때

### Step 4: 최초 인증

1. PowerShell을 열기
2. `claude` 입력 후 Enter
3. 웹브라우저가 자동으로 열리면 Claude.ai 계정으로 로그인
4. 브라우저에 "인증 성공" 메시지가 나오면 완료. PowerShell로 돌아가세요
5. 인증은 이후 자동 유지됩니다

### Step 5: 프로젝트 폴더 준비

GitHub에서 프로젝트를 내 컴퓨터로 가져옵니다.

**5-1. 프로젝트를 저장할 폴더로 이동:**

예를 들어 "문서" 폴더 아래에 저장하고 싶다면:
```
cd C:\Users\홍길동\Documents
```

> **내 경로를 모르겠다면?** 파일 탐색기에서 원하는 폴더를 열고,
> 상단 주소 표시줄을 클릭하면 `C:\Users\...` 형태의 경로가 나타납니다.
> 그것을 복사해서 `cd ` 뒤에 붙여넣으세요.

**5-2. GitHub에서 프로젝트 복제:**

1. 웹브라우저에서 GitHub 레포지토리 페이지에 접속
2. 초록색 **"<> Code"** 버튼을 클릭
3. "HTTPS" 탭이 선택된 상태에서 주소를 복사 (주소 옆 복사 아이콘 클릭)
4. PowerShell에 아래처럼 입력 (`레포지토리-주소` 부분에 복사한 주소를 붙여넣기):

```
git clone 레포지토리-주소
```

5. 처음 실행하면 **브라우저에서 GitHub 로그인 창**이 뜹니다. 본인 GitHub 계정으로 로그인하세요 (이후 자동 유지)
6. 완료되면 폴더가 생성됩니다. 해당 폴더로 이동:

```
cd 생성된-폴더-이름
```

> `ls`를 입력하면 폴더 안의 파일 목록을 확인할 수 있습니다.
> `package.json`, `src` 등이 보이면 성공입니다.

### Step 6: Git 사용자 설정 (Claude에게 요청)

이제부터 Claude Code를 사용합니다! 프로젝트 폴더 안에서 아래 명령어를 입력하세요:

```
claude
```

**Claude Code가 실행되면 이런 화면이 나타납니다:**

```
╭──────────────────────────────────────╮
│ ✻ Welcome to Claude Code!           │
│                                      │
│   /help for help                     │
╰──────────────────────────────────────╯

>
```

`>` 뒤에 커서가 깜빡이면 준비 완료입니다. 여기에 한국어로 요청을 입력하면 됩니다.

아래 내용을 입력하세요 (이름과 이메일은 본인 것으로 바꾸세요):

```
Git 사용자 설정해줘.
이름은 홍길동이고 이메일은 gildong@gmail.com이야.
```

> **왜 필요한가요?** 코드를 저장(커밋)할 때 "누가 수정했는지" 기록이 남습니다.
> 본인의 이름과 GitHub 계정 이메일을 설정하면 됩니다. 한 번만 하면 됩니다.

Claude가 명령어를 실행해도 되는지 물어보면 `y`를 입력하고 Enter를 누르세요.

### Step 7: 개발 환경 세팅 (Claude에게 요청)

Step 6에서 Claude Code가 아직 실행 중이라면 그대로 이어서 진행합니다.
종료했다면 프로젝트 폴더에서 다시 `claude`를 입력하여 실행하세요.

Claude에게 아래 내용을 전달하세요:

```
개발 환경을 세팅하고 싶어.
1. npm install 실행해줘
2. .env.local 파일을 만들어줘 (.env.local.example 참고)

환경변수 값은 내가 알려줄게:
- NEXT_PUBLIC_SUPABASE_URL: (아래 안내에 따라 확인한 값)
- NEXT_PUBLIC_SUPABASE_ANON_KEY: (아래 안내에 따라 확인한 값)
- ADMIN_EMAIL: (관리자로 사용할 Google 이메일)
```

**환경변수 값 확인 방법:**

환경변수는 서비스들을 연결하는 설정값입니다.
아래 안내에 따라 Supabase 대시보드에서 값을 찾아 Claude에게 알려주세요.

| 환경변수 | 확인 방법 |
|----------|----------|
| `NEXT_PUBLIC_SUPABASE_URL` | [Supabase 대시보드](https://supabase.com/dashboard) → 프로젝트 선택 → Project Settings → API Keys → **Project URL** 복사 |
| `NEXT_PUBLIC_SUPABASE_ANON_KEY` | 같은 페이지 → Legacy API Keys 탭 → **anon public** 키 복사 |
| `ADMIN_EMAIL` | 관리자로 사용할 본인 Google 이메일 주소 |

Claude가 명령어를 실행하거나 파일을 만들어도 되는지 물어볼 때마다 내용을 확인하고 `y`를 입력하세요.

### Step 8: 사이트 확인

세팅이 끝나면 Claude에게 이렇게 요청하세요:

```
개발 서버 실행해줘
```

Claude가 서버를 실행하면 이런 메시지가 나타납니다:

```
- Local: http://localhost:3000
```

웹브라우저 주소창에 `http://localhost:3000`을 입력하면 내 컴퓨터에서 실행 중인 사이트를 볼 수 있습니다.

> **localhost가 뭔가요?** "내 컴퓨터"를 뜻합니다. 인터넷에 공개된 것이 아니라,
> 내 컴퓨터에서만 볼 수 있는 임시 사이트입니다. 수정한 결과를 바로 확인할 때 사용합니다.

## 4. 작업별 가이드 — 누가 뭘 하나요?

모든 작업이 Claude에게 말만 하면 되는 건 아닙니다.
작업 종류에 따라 **"Claude에게 말하기"**, **"확인해서 알려주기"**, **"직접 하기"**로 나뉩니다.

### Claude에게 말만 하면 되는 작업

이 작업들은 Claude가 **혼자서 처리**할 수 있습니다. 그냥 요청하세요.

| 작업 | 요청 예시 |
|------|----------|
| 코드 수정 (디자인, 콘텐츠, 기능) | "헤더 배경색을 남색으로 바꿔줘" |
| 패키지 설치 | "이 기능에 필요한 패키지 설치해줘" |
| 개발 서버 실행 | "개발 서버 실행해줘" |
| 빌드 확인 | "빌드가 정상적으로 되는지 확인해줘" |
| 커밋 (내 PC에 저장) | "커밋해줘" |
| 푸시 (사이트에 반영) | "푸시해줘" |
| 에러 수정 | "이 에러 고쳐줘: [에러 메시지]" |
| 되돌리기 | "방금 변경한 것 되돌려줘" |

### 확인해서 알려줘야 하는 작업

이 작업들은 **대시보드에서 정보를 확인**한 뒤 Claude에게 알려주면 Claude가 처리합니다.

| 상황 | 직접 확인할 곳 | Claude에게 알려줄 내용 |
|------|--------------|---------------------|
| Git 사용자 설정 (최초 1회) | 본인 이름과 GitHub 이메일 확인 | "Git 사용자 설정해줘. 이름은 홍길동이고 이메일은 gildong@gmail.com이야" |
| 환경변수 설정 (.env.local) | [Supabase 대시보드](https://supabase.com/dashboard) → Project Settings → API Keys | "Supabase URL은 `https://xxx.supabase.co`이고 anon key는 `eyJ...`야. 이 값으로 .env.local 만들어줘" |
| 배포 에러 확인 | [Vercel 대시보드](https://vercel.com/dashboard) → Deployments → 에러 배포 클릭 | "Vercel에서 이런 에러가 나와: [에러 로그 복사]" |
| 사이트 화면 문제 | 브라우저에서 사이트 접속 | 스크린샷 붙여넣기 + "이 부분이 이상해" |
| 관리자 이메일 변경 | 새로 사용할 Google 이메일 확인 | "관리자 이메일을 newemail@gmail.com으로 변경해줘" |

### 직접 해야 하는 작업

이 작업들은 Claude가 **직접 실행할 수 없습니다.**
하지만 방법을 모르겠으면 Claude에게 물어보면 단계별로 안내해줍니다.

| 작업 | 방법 | 빈도 |
|------|------|------|
| 프로그램 설치 (Git, Node.js, Claude Code) | [3장 사전 준비](#3-사전-준비-최초-1회) 참고 | 최초 1회 |
| Anthropic 구독/결제 | [claude.ai](https://claude.ai) → 설정 → 구독 | 최초 1회 |
| GitHub 프로젝트 복제 (git clone) | [3장 Step 5](#step-5-프로젝트-폴더-준비) 참고 | 새 컴퓨터마다 1회 |
| Supabase DB 일시 중지 복구 | [Supabase 대시보드](https://supabase.com/dashboard) → 프로젝트 → Resume 클릭 | 거의 없음 |
| Vercel 환경변수 변경 | Vercel 대시보드 → Settings → Environment Variables | 거의 없음 |
| GCP OAuth 설정 변경 | [GCP 콘솔](https://console.cloud.google.com) → Auth Platform → Clients | 거의 없음 |
| 도메인 연결/변경 | Vercel 대시보드 → Settings → Domains | 거의 없음 |

> **막혔을 때:** Claude에게 "Supabase에서 API 키 찾는 방법 알려줘"처럼 물어보면
> 대시보드에서 어디를 클릭해야 하는지 단계별로 안내해줍니다.
> 서비스별 대시보드 경로는 [서비스 관리 가이드 (SERVICE_GUIDE.md)](SERVICE_GUIDE.md)에도 정리되어 있습니다.

## 5. 사용법 (매번)

사전 준비가 끝나면, 이후에는 아래 3단계만 반복하면 됩니다.

### Step 1: 터미널에서 프로젝트 폴더로 이동

PowerShell(Windows) 또는 터미널(Mac)을 열고, 프로젝트 폴더로 이동합니다:

```
cd 프로젝트/폴더/경로
```

> **폴더 경로를 잊었다면?** 파일 탐색기에서 프로젝트 폴더를 열고 주소 표시줄을 클릭하여 경로를 복사하세요.

### Step 2: Claude Code 실행

```
claude
```

`>` 표시가 나타나면 준비 완료입니다.

> **이전 대화를 이어서 하고 싶다면?** `claude --continue`를 입력하면 마지막 대화를 이어할 수 있습니다.

### Step 3: 원하는 변경 요청

한국어로 자유롭게 요청합니다.
Claude가 파일 수정이나 명령어 실행 허락을 구하면, 내용을 확인하고 `y`를 입력하세요.
작업이 끝나면 `Ctrl+C` 또는 `/exit`로 Claude Code를 종료할 수 있습니다.

## 6. 자주 하는 작업 예시

### 콘텐츠 변경

| 요청 예시 | 설명 |

## 7. 변경 후 배포

코드 수정이 완료되면 사이트에 반영해야 합니다. Claude에게 말하면 됩니다:

```
커밋하고 푸시해줘
```

이후 과정:
1. Claude가 변경 내용을 정리하여 저장(커밋)합니다
2. GitHub에 업로드(푸시)합니다
3. Vercel이 자동으로 감지하여 새 버전을 배포합니다
4. **1~2분 후** 사이트에 반영됩니다

> 배포 상태는 [Vercel 대시보드](https://vercel.com/dashboard)의 Deployments 탭에서 확인할 수 있습니다.

## 8. 실용 팁

### 8.1 요청은 "어디서 무엇을 어떻게"로 말하기

Claude는 똑똑하지만 여러분의 마음을 읽지는 못합니다. 막연하게 말하면 엉뚱한 곳을 수정할 수 있습니다.
**"어디서 + 무엇을 + 어떻게"**를 포함하면 한 번에 원하는 결과를 얻을 확률이 크게 높아집니다.

| 안 좋은 요청 | 좋은 요청 |
|-------------|----------|
| "색상 바꿔줘" | "메인 페이지 헤더의 배경색을 좀 더 진한 남색으로 바꿔줘" |
| "글씨가 작아" | "블로그 글 목록에서 글 제목의 글씨 크기를 지금보다 좀 더 키워줘" |
| "링크 추가해줘" | "푸터 연락처 영역에 인스타그램 링크(https://instagram.com/myid)를 추가해줘" |
| "디자인 고쳐줘" | "모바일에서 블로그 카드가 좌우로 잘리는데, 화면 너비에 맞게 조정해줘" |
| "버그 있어" | "글 상세 페이지에서 뒤로가기 버튼을 누르면 메인이 아니라 빈 화면이 나와" |

### 8.2 한 번에 하나만 요청하기

"헤더 색상 바꾸고, 푸터에 링크 추가하고, 글 목록 정렬도 변경해줘"처럼 여러 가지를 한꺼번에 요청하면,
하나가 잘못되었을 때 어떤 변경이 문제인지 찾기 어렵습니다.

**권장 흐름:**
1. 하나의 변경을 요청한다
2. Claude가 수정을 완료하면 결과를 확인한다
3. 괜찮으면 "커밋해줘"로 저장한다
4. 그 다음 변경을 요청한다

> 요리할 때 한 가지 재료를 넣고 맛을 본 후 다음 재료를 넣는 것과 같습니다.
> 한꺼번에 다 넣으면 무엇이 맛을 망쳤는지 알 수 없습니다.

### 8.3 스크린샷으로 보여주기

"좀 더 예쁘게", "간격을 적당히"같은 표현은 사람마다 느낌이 다릅니다.
원하는 디자인의 스크린샷이나 참고 이미지를 보여주면 Claude가 훨씬 정확하게 구현합니다.

- 스크린샷을 클립보드에 복사(`Ctrl+C`) 후, Claude Code 대화창에 `Ctrl+V`로 붙여넣기
- 현재 문제가 있는 화면을 캡처해서 "여기서 이 부분을 수정해줘"
- 다른 사이트에서 마음에 드는 디자인을 캡처해서 "이런 느낌으로 바꿔줘"

### 8.4 수정 후 반드시 눈으로 확인하기

Claude가 "수정 완료했습니다"라고 해도, 실제 화면에서 의도대로 보이는지는 직접 확인해야 합니다.

1. 개발 서버가 꺼져 있다면 "개발 서버 실행해줘"라고 요청
2. 브라우저에서 `http://localhost:3000` 접속하여 확인
3. **PC와 모바일 양쪽에서** 확인 (PC 브라우저에서 `F12` → 상단의 모바일 아이콘 클릭)
4. 문제가 있으면 구체적으로 알려주기: "왼쪽 여백이 너무 넓어" 또는 "글씨가 배경과 겹쳐서 안 보여"

### 8.5 대화가 길어지면 `/clear`로 새로 시작하기

Claude는 대화가 길어질수록 앞의 내용을 잊거나 헷갈릴 수 있습니다.
하나의 작업이 끝나면 대화를 정리하고 새로 시작하는 것이 좋습니다.

**`/clear`를 쓰면 좋은 때:**
- 하나의 수정 작업이 완료되고 커밋까지 했을 때
- Claude가 같은 실수를 반복할 때
- 전혀 다른 작업을 시작할 때

### 8.6 되돌리기는 언제든 가능합니다

수정 결과가 마음에 들지 않거나 사이트가 이상해졌을 때, 당황하지 않아도 됩니다.

**방법 1 — Claude에게 말하기 (가장 간단):**
```
방금 변경한 것 되돌려줘
```

**방법 2 — `Esc` 키 두 번 누르기 (더 확실한 방법):**
1. `Esc` 키를 빠르게 두 번 누르면 "되감기 메뉴"가 나타남
2. 되돌아가고 싶은 시점을 선택
3. 코드와 대화가 모두 그 시점으로 돌아감

> **핵심:** 커밋(저장) 전에는 쉽게 되돌릴 수 있으므로, 수정 결과를 확인하기 전까지는 커밋하지 마세요.

### 8.7 "커밋"과 "푸시" — 꼭 알아야 할 두 단어

이 두 단어는 사이트 관리에서 계속 나오는 개념입니다.

| 용어 | 비유 | 의미 |
|------|------|------|
| **커밋(commit)** | 문서에 "저장" 버튼 누르기 | 변경 내용을 내 컴퓨터에 기록. 언제든 이 시점으로 돌아올 수 있음 |
| **푸시(push)** | 저장한 문서를 클라우드에 올리기 | 내 컴퓨터의 기록을 GitHub(인터넷 저장소)에 업로드 |
| **배포(deploy)** | 클라우드 문서가 자동으로 웹사이트에 반영 | 푸시하면 Vercel이 자동으로 사이트를 업데이트 |

**중요한 순서:**
```
수정 → 확인 → 커밋(내 PC에 저장) → 푸시(인터넷에 업로드) → 자동 배포(사이트 반영)
```

> "커밋하고 푸시해줘"라고 한 번에 말하면 편하지만,
> **수정 결과를 먼저 확인한 후에** 이 말을 하세요.
>
> 더 많은 Git 명령어의 의미가 궁금하다면 [부록: Git 명령어 이해하기](#부록-git-명령어-이해하기)를 참고하세요.

### 8.8 Claude가 "허락"을 구할 때 꼼꼼히 읽기

Claude는 파일을 수정하거나 명령어를 실행하기 전에 허락을 구합니다.
이때 무심코 "Y"를 누르지 말고, **어떤 파일을 어떻게 바꾸려는지** 확인하세요.

- Claude가 수정하려는 파일 이름이 예상한 파일이 맞는지 확인
- "N개의 파일을 수정합니다"에서 N이 너무 크면(5개 이상) 한번 의심하기
- 잘 모르겠으면 `n`(아니오)을 누르고 "어떤 파일을 왜 수정하려는 건지 설명해줘"라고 질문

> 집 수리 기사가 "이 벽을 뚫어도 될까요?"라고 물을 때, 내용을 확인하지 않고 "네"라고 하면 안 되는 것과 같습니다.

### 8.9 큰 변경은 작은 단계로 나누기

"메인 페이지 전체 디자인을 바꿔줘"처럼 큰 요청은 Claude가 수많은 파일을 한꺼번에 수정하게 됩니다.
결과가 마음에 들지 않을 때 되돌리기도 복잡해집니다.

큰 변경을 원할 때는 Claude에게 먼저 계획을 세우라고 요청하세요:
```
메인 페이지 디자인을 전체적으로 바꾸고 싶어.
먼저 어떤 순서로 작업하면 좋을지 계획을 세워줘. 아직 코드는 수정하지 마.
```

Claude가 계획을 보여주면:
1. 계획이 괜찮은지 확인
2. "1단계부터 진행해줘"로 하나씩 요청
3. 각 단계마다 확인 후 다음 단계 요청

### 8.10 에러 메시지가 나오면 그대로 복사해서 알려주기

"에러가 나요"만 말하면 Claude가 원인을 찾기 어렵습니다.
화면에 나타난 에러 메시지를 그대로 보여주면 훨씬 빠르고 정확하게 해결됩니다.

```
사이트에 이런 에러가 나와:
[에러 메시지를 여기에 붙여넣기]
이 에러를 고쳐줘.
```

또는 에러 화면을 **스크린샷으로 캡처**해서 `Ctrl+V`로 붙여넣기도 좋습니다.

> **안 좋은 예:** "사이트가 안 돼요"
> **좋은 예:** "블로그 글을 클릭하면 흰 화면만 나오고 아무것도 안 보여"

### 8.11 Claude가 멈추거나 이상할 때

| 상황 | 대처법 |
|------|--------|
| Claude가 멈추거나 너무 오래 걸릴 때 | `Esc` 키를 한 번 눌러서 중단시키기 |
| 같은 실수를 반복할 때 | `/clear` 입력 후, 더 구체적으로 다시 요청 |
| Claude Code 자체가 응답하지 않을 때 | 터미널 창을 닫고 다시 `claude` 실행 |
| 수정 결과가 완전히 잘못되었을 때 | `Esc` 두 번 → 되감기 메뉴에서 이전 상태 선택 |

> `Esc` = 멈추기, `Esc` + `Esc` = 되돌리기, `/clear` = 새로 시작

### 8.12 배포 전에 빌드 확인하기

개발 서버(`localhost:3000`)에서는 잘 보여도, 실제 배포할 때 에러가 발생할 수 있습니다.
푸시 전에 Claude에게 이렇게 요청하세요:

```
빌드가 정상적으로 되는지 확인해줘
```

- **성공:** 안심하고 "커밋하고 푸시해줘" 요청
- **실패:** Claude에게 "빌드 에러 고쳐줘"라고 하면 자동으로 수정

> **권장 흐름:** 수정 → 눈으로 확인 → 빌드 확인 → 커밋 → 푸시

### 8.13 이전 대화 이어서 하기

어제 작업하던 것을 오늘 이어서 하고 싶을 때, 처음부터 다시 설명하지 않아도 됩니다.

```
claude --continue
```

최근 대화 목록에서 골라서 이어하려면:
```
claude --resume
```

> 이전 대화가 짧고 단순했다면 `--continue`, 길고 복잡했다면 새로 시작하는 것을 추천합니다.

## 9. 알아두면 좋은 명령어

| `Esc` | Claude 작업 중단 | Claude를 멈추고 싶을 때 |
| `Esc` + `Esc` | 되감기 메뉴 열기 | 이전 상태로 되돌아가고 싶을 때 |

## 부록: Git 명령어 이해하기

Claude Code를 사용하다 보면 Claude가 `git ○○○` 명령어를 실행해도 되는지 허락을 구할 때가 있습니다.
이 명령어들이 무엇을 하는 건지 알아두면 안심하고 승인할 수 있습니다.

> Git은 **코드의 변경 이력을 관리하는 도구**입니다.
> 문서 작업에서 "버전 기록"이나 "변경 내역 추적"과 비슷한 역할을 합니다.
> Claude가 명령어를 대신 실행해 주므로 외울 필요는 없지만, 어떤 작업인지 알아두면 안심하고 승인할 수 있습니다.

### 자주 보게 되는 명령어

#### 변경 확인 계열

| 명령어 | 비유 | 하는 일 |
|--------|------|---------|
| `git status` | "지금 뭐가 바뀌었지?" 확인하기 | 어떤 파일이 수정/추가/삭제되었는지 목록을 보여줌 |
| `git diff` | 수정 전과 후를 나란히 놓고 비교하기 | 파일에서 정확히 어느 부분이 바뀌었는지 보여줌 |
| `git log` | 수정 일지 넘겨보기 | 지금까지의 저장(커밋) 기록을 시간순으로 보여줌 |

#### 저장 계열

| 명령어 | 비유 | 하는 일 |
|--------|------|---------|
| `git add 파일명` | 저장할 파일을 골라서 바구니에 담기 | "이 파일의 변경 사항을 저장 대상에 포함시켜줘" |
| `git commit -m "설명"` | 바구니에 담은 것들을 묶어서 저장하기 | 변경 내용을 내 컴퓨터에 기록. "설명"에는 무엇을 바꿨는지 메모 |
| `git push` | 저장한 기록을 인터넷(GitHub)에 올리기 | 내 컴퓨터의 기록을 GitHub에 업로드 → Vercel이 자동 배포 |

#### 되돌리기 계열

| 명령어 | 비유 | 하는 일 |
|--------|------|---------|
| `git checkout 파일명` | 수정하기 전 원본으로 되돌리기 | 해당 파일을 마지막 저장 상태로 복원 |
| `git stash` | 작업 중인 것을 잠시 서랍에 넣어두기 | 수정 중인 내용을 임시 보관하고 깨끗한 상태로 만듦 |
| `git stash pop` | 서랍에서 다시 꺼내기 | 임시 보관했던 수정 내용을 다시 불러옴 |

#### 브랜치 계열

| 명령어 | 비유 | 하는 일 |
|--------|------|---------|
| `git branch` | 현재 어떤 작업 공간에 있는지 확인 | 브랜치(작업 공간) 목록과 현재 위치를 보여줌 |
| `git checkout -b 이름` | 새로운 작업 공간을 만들어서 이동 | 원본을 건드리지 않고 별도 공간에서 작업할 수 있음 |
| `git merge` | 별도 작업 공간의 결과를 원본에 합치기 | 다른 브랜치에서 작업한 내용을 현재 브랜치에 합침 |

### 전체 흐름 이해하기

```
[내 컴퓨터]                          [인터넷]

 파일 수정                            GitHub
    ↓                                  ↑
 git add (저장할 파일 선택)              │
    ↓                                  │
 git commit (내 PC에 기록)              │
    ↓                                  │
 git push ──────────────────────────→ 업로드
                                       ↓
                                    Vercel이 감지
                                       ↓
                                    자동 배포
                                       ↓
                                    사이트 반영
```

### Claude가 이런 명령어를 실행하려고 할 때

Claude가 허락을 구하면 이렇게 판단하세요:

| Claude가 하려는 것 | 안전한가요? | 승인 기준 |
|-------------------|-----------|----------|
| `git status`, `git diff`, `git log` | 매우 안전 | 그냥 확인만 하는 것이므로 항상 승인 |
| `git add`, `git commit` | 안전 | 내 컴퓨터에만 저장되므로 승인. 커밋 메시지가 변경 내용과 맞는지 확인 |
| `git push` | 주의 | 사이트에 반영됨. 수정 결과를 확인한 후에만 승인 |
| `git checkout`, `git stash` | 안전 | 되돌리기/임시 보관이므로 승인 |
| `git branch`, `git checkout -b` | 안전 | 새 작업 공간을 만드는 것이므로 승인 |

> **요약:** `push`만 신중하게 판단하면 됩니다. 나머지는 내 컴퓨터 안에서만 일어나는 작업이므로 안전합니다.
